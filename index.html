CSS

語法

  • 使用 4 個空格的 soft tabs
    ─ 這是唯一可以保證程式碼在任何環境下，render 結果一致的方法。
  • 幫選擇器分組時，獨立的選擇器自成一行。
  • 左大括號左邊留 1 個空白來提升可讀性。
  • 右大括號應另起新行。
  • 每條宣告裡的 : 後面留一個空白。
  • 每條宣告應自成一行，錯誤會回報的更精準。
  • 每條宣告用分號結束。
  • 以逗號分隔的特性，逗號後要加一個空白
    （比如右例的 box-shadow，inset前方逗號後有留空白）。
  • 小數值簡寫成 .5，不要寫成 0.5。
  • hex 值全小寫，如：#fff。
    在快速掃過文件時，小寫符號比大寫符號更容易分辨，因為形狀較為不同。
  • 盡可能使用 hex 值的簡寫，比如用 #fff 來取代 #ffffff。
    色碼盡量不使用 英文縮寫，除非在 preprocessor 裡使用 color function 時的簡寫。
  • 選擇器中的屬性值要用雙引號包起來，比如 input[type="text"]。
    屬性值只有某些情況下可省略，加引號的一致性更好。
  • 0 不用寫單位。寫成 margin: 0; 而不是 margin: 0px;。

對於這裡使用的術語有任何問題嗎？請參考維基百科上關於層疊樣式表條目裡的語法小節。

宣告順序

相關的屬性宣告應以下列順序分組：

  1. Content
  2. Positioning
  3. 顯示 & Box Model
  4. Typography
  5. Visual
  6. Misc
  7. Animation

Content: 內容本身最重要，需定義在最前面較清楚(例如：偽元素內容)。

Positioning: 可以從正常文件順序裡移除元素(out of flow)的位置宣告，或是覆寫牽扯到 Box Model 的樣式。

顯示 & Box Model: 決定元件的位置與尺寸 或 與 Box Model 相關。

其它只在元件內部起作用的屬性放後面，因為這不會影響前面。

Animation: 動畫因為較不影響排版，所以放最後。

宣告原則

  1. 重要性或影響性高的屬性放前面，例如：Positioning 區塊 放 Box Model 區塊前面。
  2. 相依性的屬性要排序，例如：position 放 top 前面。
  3. 相關性高的放一起，例如：flex 相關屬性。

關於屬性順序的完整列表，請參考 Recess、idiomatic-css #declaration-order。

單行宣告

當一組規則只包含單條宣告的的情況裡，考慮移除換行；寫成單行的可讀性更高、更容易編輯。任何包含多條宣告的一組規則應該要分為多行。

這麼分的關鍵因素是錯誤偵測─比如：CSS Validator 表示 183 行有語法錯誤。如果是單行宣告便就是那行，多條宣告沒分行的話則會找到抓狂。

簡寫記法

宣告盡量少用簡寫，最好明確的將所有的值寫出來。過度濫用簡寫的特性包含：

  • padding
  • margin
  • font
  • background
  • border
  • border-radius

通常我們只需設定需要的值，簡寫會設定到多餘的值。舉例來說，HTML 標題只會設定 top 與 bottom margin，所以只要更改這兩個值即可。濫用特性縮寫只會寫出更差的程式碼，也會有無謂的覆寫與不預期的副作用。

註解

程式碼是由人來撰寫與維護的。
確保程式碼精準描述、有良好的註解，讓別人看起來是很親切。
好的註解傳遞意圖、意境。
註解不要只是重複元件或是 Class 的名稱。

長註解記得使用完整的句子，一般的筆記用簡潔的用語。

{% highlight css %}{% include css/comments.css %}{% endhighlight %}

Class 名稱

  • 採用 BEM naming, 兩個下底線代表階層，兩個連接號代表狀態
  • Class 名稱為小寫，非 BEM 規則時盡量使用連接號，dash。（不要使用底線或 camelCase）。
    連接號作為相關 Class 的分類符。（像是 .btn 和 .btn-danger）。
  • 避免使用過短、隨便的縮寫。 button 簡寫為 .btn 很好，但 .s 不代表任何元素。
  • 盡量保持 Class 簡潔有力。
  • 使用有意義的名字；使用有意義、能說明結構的名稱，而不是外表看起來的名稱。
  • 根據最近的父類、基類加上前綴。
  • 使用 .js-* Class 來標示行為（相對樣式來說），這些類別放在另外的 CSS。
{% highlight css %}{% include css/class-names.css %}{% endhighlight %}

選擇器

  • 就 render 的性能來說，使用 Class 會比通用的元素標籤好。
  • 重複出現的元件，避免使用多個屬性選擇器（比如：[class^="..."]）。大家都知道這會影響到瀏覽器的性能。
  • 保持選擇器簡短，每個選擇器盡量不要超過 3 個元素。
  • 只在需要的情況下使用後代 Class （比如沒有使用前綴 class 的情況）。

最佳實踐

HTML

推薦閱讀 Essential Image Optimization 深入瞭解圖片的最佳化技巧

  • 使用 HTML5 語義化標籤，例如 header, section, footer, main, nav
  • 外部超連結 (target="_blank") 加上 rel="noopener"，避免把母視窗的 JS 資料帶出，確保 window.opener 為 null，增進安全性也改善效能 [深入閱讀]

Images

推薦閱讀 Essential Image Optimization 深入瞭解圖片的最佳化技巧

  • 把圖片 commit 進 repository 之前，應確實進行壓縮作業，推薦使用 ImageOptim、Imagemin 等工具；SVG 使用SVGOMG
  • 多張零碎的小圖可以合併成 sprite (雪碧圖) ，若專案基礎建設支援 HTTP/2 則免
  • img 標籤指定 width 與 height 屬性避免瀏覽器載入圖片完成後才進行 layout 造成內容抖動影響體驗
  • 圖片應有 alt 屬性描述其意義，純裝飾圖可填入空字串或是適合的符號，推薦閱讀 Alt-texts: The Ultimate Guide 替代文字撰寫指南
  • 若頁面會載入許多圖片，建議加上 Lazy loading 機制，節省不必要的流量開銷以及改善閱讀體驗

JavaScript

V8 6.0 引擎的新架構 TurboFan 對 ES2015 的語法有更好的效能 資料來源

  • 使用明確的物件比較，在 V8 6 引擎平均增快 15%
  • 使用 ES2015 的其餘參數 (Rest parameters) 特性，相較於舊式語法實作等效功能，平均增快 4.5 倍
  • 為防止外部 JavaScript 堵塞關鍵渲染路徑，請使用 defer 屬性延後剖析執行 JavaScript，或採用 async 非同步載入，如果載入的外部 JS 具有相依性 (例如需要 jQuery)，請務必確認正確的順序載入。 [參考閱讀]
  • 避免使用 document.write() 載入外部 script，會導致堵塞關鍵渲染路徑，Chrome 在緩慢的網路連線環境下會封鎖此類連線之載入 [資料來源]

編輯喜好

使用下列設定來設定編輯器，避免掉常見的程式碼不一致和醜陋的 diffs：

  • 使用四個空白的 soft-tabs。
  • 存檔時抹除行尾空白。
  • 編碼設定為 UTF-8。
  • 檔案最後加入一空白行。

考慮將以上偏好應用到專案的 .editorconfig 檔案並撰寫文件說明。舉個例子，參考 Bootstrap 的 .editorconfig。瞭解更多內容，請參考 editorconfig.org。
